# Harness Engineering 的下一步：用 Fitness Function 定义 AI Agent 的完成条件

## 当 Harness Engineering 开始落地，问题就不再只是“怎么生成代码”

过去一段时间，Harness Engineering 之所以会成为一个热门话题，本质上是因为大家逐渐意识到，AI Coding
的问题从来不只是模型能力问题。模型当然重要，上下文工程、提示词策略、工作流拆分、多 Agent 协作也都重要，但这些讨论大多仍然停留在生成侧。它们回答的是：系统如何让
AI 更稳定地写代码、更顺畅地理解仓库、更高效地分解任务。然而，一旦 AI Agent 真正进入软件交付流程，另一个问题很快就会浮现出来，而且往往比生成本身更棘手：系统究竟如何判断，这个
Agent 已经完成了任务。

这是一个过去在人工开发场景中很少被显式讨论的问题，因为“完成”长期以来都被包裹在人类经验里。开发者写完代码，跑过测试，看过功能，提交
Pull Request，经过 code review，再通过 CI、集成验证与上线观察，团队最终形成一种共识：这次改动大体是可信的。这个过程并不总是严格，也不一定完全形式化，但它背后始终有人的责任判断在兜底。可一旦软件开发进入
Agent Loop，这种默认前提就不再成立了。Agent
可以很快生成代码，也可以很快修掉一个报错，还可以在局部任务上表现得相当优秀，但它同样会很快制造另一类结果：代码看起来已经完成，实际上却只是完成了一半，甚至连最重要的那一半都没碰到。

这类问题并不抽象。功能路径可以跑通，不代表负向路径被覆盖；接口实现已经修改，不代表契约同步了；测试数量增加，不代表关键不变量得到了验证；某个模块局部正确，也不代表多个实现之间的语义仍然一致。过去这些问题靠经验、靠
review、靠“谁熟悉这块系统”来补位，而在 AI 参与开发之后，这些隐性补位机制会迅速失效。也因此，Harness Engineering
真正进入工程实践之后，焦点就会从“如何让 Agent 生成代码”逐渐转向“系统如何定义完成条件”。换句话说，Harness Engineering
的下一步，不是继续围绕生成侧做局部优化，而是把“完成”这件事从经验判断转化为可执行、可审计、可阻断的工程信号。

## Fitness Function 在 AI 时代，不再只是架构概念

Fitness Function
原本并不是一个新概念。它来自演进式架构，用来持续验证系统是否满足某些架构特征，例如依赖方向、性能指标、可用性约束或安全要求。这个概念过去更多被用于架构治理语境里：我们用自动化机制不断确认系统有没有偏离设计意图，有没有突破原本设定的边界条件。但在
AI Agent 开始参与开发之后，Fitness Function 的角色发生了一个很微妙、但非常重要的变化。它不再只是“架构质量检查”的工具，而开始成为一种新的完成条件机制。

这里的关键变化在于，AI Agent
并不天然理解“什么叫真正做完”。它会把自己能观察到的局部信号当成完成依据，例如命令执行成功了、某个测试绿了、报错消失了、代码文件已经被改动了。可对于真实的软件交付来说，这些局部成功从来不等价于整体完成。完成不是“写了代码”，也不是“解决了眼前报错”，而是改动已经满足项目中那些真正不可妥协的工程条件。于是，Fitness
Function 在这里承担的新角色，就是把这些条件重新编码成 Agent 能消费的形式。它不是告诉 Agent
“你应该尽量写得更好”，而是明确告诉系统“哪些信号一旦没有出现，这个任务就不能被视为完成”。

这也是为什么我觉得，Fitness Function 在 AI
时代值得被重新讨论。它真正重要的地方，并不在于它能让我们“检查更多”，而在于它能帮助系统重新建立一种可信的完成判断。过去这个判断主要掌握在人和团队经验手里，现在它必须逐渐被沉淀到仓库和交付链路里。也正因为如此，Fitness
Function 不再只是一个演进式架构术语，而开始成为 Harness Engineering 的核心部件：它决定的不只是系统有没有偏离架构，而是
Agent 在什么条件下才被允许退出循环。

## Routa 不是一个示例项目，而是我们对这个问题的工程回答

为了验证这件事，我们在 Routa 里并没有去做一个抽象的平台，而是尝试直接在代码库中构建一套完整的 Fitness 架构。与其说 Routa
是一个“用了 Fitness Function 的项目”，不如说它是我们对这个问题的一次工程回答：如果要让 AI Agent
的完成条件真正变成工程信号，代码库内部应该长成什么样。

Routa 的一个基本判断是，Fitness 不能只停留在理念层面，也不能只存在于某个外部治理平台里。它必须是仓库的一部分，必须能够被
Agent 读取，被脚本执行，被 hooks 和 CI 消费，并在失败时明确阻断流程。也就是说，Fitness
不是一份额外的质量附录，而是整个开发与交付流程中决定任务能否结束的那层结构。围绕这个判断，我们把 Fitness
拆成了几个相互协作的层次：入口协议、规则文档、证据账本、统一执行器、契约检查以及交付链路中的门禁机制。它们听起来并不复杂，但组合在一起之后，作用就很明确：让“完成”不再由
Agent 自己解释，而由仓库中的工程信号决定。

整个入口从 AGENTS.md 开始。这个文件并不试图解释整个项目，也不打算在开头就把所有背景知识灌给 Agent。相反，它只定义最小执行边界：代码变更之后必须运行
Fitness 检查；某些任务需要按指定顺序阅读特定规则文档；调试阶段允许临时引入 console.log，但修复后必须清理；提交必须保持
baby-step，而不是一次性提交一个 kitchen sink change。这样的设计其实很有代表性，因为它对应了一个经常被忽略的事实：对于 Agent
来说，入口比说明更重要。它不需要一上来理解整个世界，它需要先被带到正确的位置。AGENTS.md 在这里扮演的不是百科全书，而是导航协议。它把
Agent 从“开始执行任务”引导到“进入 Fitness 规则系统”，再引导到具体的证据文件和执行器。这个过程本质上就是一种渐进式披露：规则不是一次性塞进上下文，而是被拆成可导航、可按需加载的知识模块。

## 规则必须可读，也必须可执行

但仅仅有入口导航，还不足以支撑一个真正可运行的 Fitness 系统。更关键的是，规则如何表达。我们在 Routa 里做的一个核心选择，是没有把规则写死在
CI 配置、平台控制台或者某套外部 DSL 中，而是直接写进 Markdown 的 frontmatter
里。这样做的理由非常现实：如果规则只对机器友好，那么团队无法自然维护；如果规则只对人友好，那么系统无法统一执行。前者会造成治理平台和代码库脱节，后者则会让所有规则重新退化为“口头共识”。

把规则写在 Markdown frontmatter 中，等于同时保留了两件事。第一，它仍然是文档的一部分，开发者和 Agent
都可以阅读，团队可以像维护普通文档一样维护它。第二，它又具备明确的结构化特征，脚本可以直接解析
dimension、weight、metrics、hard_gate 等字段，并根据这些字段执行具体命令。于是，规则既是可阅读知识，也是可执行声明。新增一个
Fitness 维度，本质上只是在 docs/fitness 目录下新增一个带 frontmatter 的 Markdown 文件，而不是去修改一套复杂的治理平台。这个特点非常重要，因为它使得
Fitness 规则能够跟着项目一起演进，而不是被外置平台锁死在某个治理团队手里。

这种设计背后其实也延续了我之前在 AI-Friendly Code Structure 里写过的一个判断：对 AI
友好的系统，不能只追求“信息存在”，而要追求“信息能被低成本发现与消费”。把 Fitness
规则变成代码库中的结构化文档，本质上就是让这套质量约束成为仓库里的显式知识，而不再依赖少数人的记忆和经验。对于 Agent
来说，这一点尤其关键。它无法像资深工程师那样自动补全那些隐性共识，因此，系统必须把这些共识显式化，并且用一种 Agent
真正能读取和执行的形式表达出来。

## 真正重要的不是规则本身，而是证据

不过，规则声明还只是第一层。一个真正可靠的 Fitness 系统，不仅需要告诉 Agent
应该检查什么，还需要告诉系统：这些检查的状态是什么，哪些问题已经被验证，哪些仍然悬而未决，哪些风险只是暂时被搁置。也正因为如此，Routa
在规则之外，还引入了证据文件这个层次。

这些证据文件并不是普通的测试说明书，也不只是“待办事项列表”。它们更像是一种工程账本，记录的是验证本身的状态。哪些场景已经
VERIFIED，哪些仍然 TODO，哪些因为外部依赖或当前条件限制被标记为
BLOCKED，哪些失败模式在项目里高频出现，都会明确写在这些文档里。这样一来，文档的角色就发生了变化。它不再只是“告诉人应该怎么做”，而开始承担“为系统提供可审计状态”的职责。对于
AI Agent 来说，这一点特别重要，因为它让代码库中的历史经验第一次以一种更稳定的方式被保存下来。系统不只是记住“这里以前出过问题”，而是明确记住“这种问题应该如何被重新验证，哪些部分已经被验证，哪些部分仍然存在风险”。

从架构治理的角度来看，这其实也是一个很有意思的变化。过去我们习惯把风险清单、技术债台账和演进路线图视为人类治理材料，它们更多服务于复盘、规划和协调。而在
Agent 时代，这些东西如果不能被重新编码成机器可消费的信息，就很难真正进入自动化开发流程。Routa
的证据文件，某种程度上正是在做这件事：把原本只对人有意义的经验材料，重新转译成 Agent 和执行器都能理解的验证上下文。这样，Fitness
不再只是“规则”，而开始形成一种对 Agent 友好的工程记忆。

## 执行器的意义，在于收回规则解释权

当规则和证据都存在之后，接下来的问题就是：谁来解释它们？在很多团队里，最危险的恰恰是这一步。规则虽然写下来了，但真正执行时总会出现模糊空间；文档虽然存在，但是否更新、是否被认为充分，依旧要靠人的经验判断。结果是，系统表面上看起来已经有了一套治理结构，实际上关键时刻仍然依赖“谁更有话语权”来决定什么算通过，什么算暂时放过。

Routa 里那个看上去并不复杂的 fitness 执行脚本，真正重要的价值就在这里。它扫描 docs/fitness 下的 Markdown，解析 YAML
frontmatter，逐项执行命令，根据输出模式或退出码判定通过与否，汇总维度结果，并在 Hard Gate
失败时立即阻断流程。它的复杂度不高，但它做了一件特别关键的事：把规则解释权从人的经验中收回来，重新交给统一执行器。系统不再接受“这次应该问题不大”“warning
先留着”“契约大体兼容”这种模糊表述，而只接受规则中声明过的命令、可观察的输出和门禁结果。

这一步对 Agent 来说尤其重要，因为 Agent
天然倾向于乐观解释局部成功。只要某个报错消失了，或者某个命令绿了，它就很容易把这理解为任务已经接近完成。而统一执行器的作用，就是不断把这种乐观偏差重新压回到工程信号层面：不是你觉得改完了，而是系统能不能证明你已经满足了退出条件。某种意义上，这正是
Harness Engineering 的核心精神之一：不要让模型自己决定什么时候结束循环，而要让仓库中的工程结构来决定。

## 多实现系统里，最危险的不是 bug，而是语义漂移

如果说前面这些机制主要解决的是“完成如何被表达、被记录、被执行”，那么在 Routa 里，最有代表性的部分其实是契约一致性检查。因为
Routa 本身是一个双后端系统，既有 Next.js API 路由，也有 Rust/Axum 的实现。在这种多实现结构里，AI Agent 最容易制造的问题往往不是单点
bug，而是语义漂移。也就是说，某个局部修改本身也许是对的，但多个实现之间开始悄悄失去同构关系。

这类问题在人工开发里已经不罕见，在 Agent 参与修改之后则会被进一步放大。前端 route 改了，Rust 端没改；Rust 返回结构变了，OpenAPI
契约没更新；契约本身已经修改，但测试只覆盖了一个实现。这些都不是传统意义上的“写错代码”，却是对系统稳定性极具杀伤力的问题，因为它们破坏的是多个实现之间本应共享的语义中心。过去这类问题常常依赖熟悉系统的人在
review 时发现，但如果系统越来越多地依赖 Agent 做局部修复和增量变更，就不能再指望这种隐式补位机制稳定存在。

也正因为如此，我们把 API Contract 放到了 Fitness 的核心位置。OpenAPI 文件被当作单一事实来源，要求契约优先于实现变更，要求
Next.js 与 Rust 两侧围绕同一组 endpoint 和数据语义收敛，再通过检查脚本去同时解析契约、前端路由和 Rust 路由，输出缺失项、不一致项以及没有回写到
contract 的实现。这个设计真正重要的地方，不只是“接口检查更全面了”，而是它把多实现之间的语义一致性，从 code review
里的经验项，提升为一个可执行的 Fitness 检查项。对于 Agent 时代的软件工程来说，这一点非常关键，因为 AI
擅长修局部，却并不擅长自动维护多个实现之间长期稳定的同构关系。契约优先，本质上是在给 Agent 提供一个不容易漂移的重心。

## Hard Gate 才是真正定义“完成”的地方

很多团队在谈质量治理时，喜欢设计各种评分机制。指标越多、分数越高，系统看上去就越健康。这种方式在人工开发环境里有一定价值，因为分数可以帮助团队看到趋势、定位薄弱项、推动改进。但在
AI Agent 场景下，单纯的评分体系是不够的。原因很简单：Agent 天然会把“还不错”误解成“可以结束”。如果系统只有健康分数，没有明确的门禁条件，那么局部成功就会不断被误判成整体完成。

所以，在 Routa 里，Fitness 最终并不是一个评分系统，而是一个阻断系统。测试、契约检查、lint、安全扫描等维度都可以成为 Hard
Gate，只要其中任意一项失败，任务就不应被视为完成。这个区分非常重要，因为它把“质量折损”和“流程终止”明确区分开来。有些问题可以被记录下来以后持续改进，但有些问题只要存在，就意味着当前这次改动不能继续往前推进。对于
Agent 来说，这样的边界尤其必要。因为 Agent 并不会天然理解“这类风险只是需要注意，那类风险则意味着必须停下”。系统必须把这种判断直接编码进执行逻辑里。

从这个角度看，Hard Gate 实际上就是 Agent 时代的 Definition of
Done。它回答的不是“这段代码看起来怎么样”，而是“在什么条件下，这个自动化参与者被允许退出循环”。而一旦把这个问题说清楚，Fitness
Function 的意义也就发生了变化。它不再只是“质量保障的一部分”，而成为 AI Agent 是否真正完成任务的判定机制。换句话说，Harness
Engineering 如果要真正落地，最后一定会走到 Hard Gate 这一层：没有门禁，完成条件就仍然是模糊的；而没有可执行的完成条件，Agent
Loop 就永远不够可信。

## 从 Routa 回看，我们真正想回答的是什么

所以，如果把 Routa 放回更大的方法论背景里去看，它并不是一个单独的项目实践，而更像是我们在回答一个越来越现实的问题：当软件开发中出现越来越多自动化参与者时，系统究竟凭什么相信，它已经真的完成了任务？

这个问题和我之前几篇文章其实是连在一起的。ArchGuard Pipeline 讨论的是如何把架构治理嵌入流水线；Architecture
Self-Governance 讨论的是如何让治理从人工评审走向系统内生机制；AI-Friendly Code Structure 讨论的是如何把仓库组织得更适合
AI 理解与导航。而 Routa 的 Fitness 架构，恰好把这几条线收束到了一起。它既是一种流水线中的质量治理机制，也是一种仓库内部的自治理结构，同时还是一种专门面向
Agent 的完成条件系统。

从这个角度说，我更愿意把它理解为 Harness Engineering 的下一步。因为当大家还在热烈讨论上下文、提示词和协作模式时，真正影响 AI
Coding 能否进入真实交付环境的，往往不是生成链路本身，而是系统是否拥有一套可信的退出条件。没有这套退出条件，再强的生成能力也只能带来更快的试错；有了它，Agent
才开始真正具备进入工程系统的资格。

## 结语：AI 时代的软件工程，需要重新发明“完成”

当越来越多的代码开始由 AI Agent
生成、修改与修复时，软件工程真正面临的变化，并不只是“写代码的人变了”，而是“完成这件事的判定方式变了”。过去，我们更关心代码怎么被写出来；现在，我们必须更明确地回答：代码什么时候才算真正完成。

Routa 的实践给我们的启发并不在于它用了多复杂的平台，也不在于它列了多少质量维度。它最值得借鉴的地方，是它把这些维度真正嵌入了开发和交付流程：用
AGENTS.md 提供入口导航，用 Markdown frontmatter 声明规则，用证据文件记录验证状态，用统一执行器收敛规则解释，用契约检查约束多实现一致性，再用
hooks 与 CI 把这一切接进完整的交付链路。它的复杂度不高，但工程意味很强。它让 Agent 的完成条件不再由模型自己解释，而由仓库中的工程信号决定。

这也许正是 Harness Engineering 在今天最值得继续推进的方向。它不只是约束生成，不只是管理上下文，也不只是为 Agent
提供若干辅助工具。它最终要解决的，是软件工程里那个最根本的问题之一：当自动化参与者越来越多时，系统究竟如何重新定义“完成”。而
Fitness Function，正在成为这个问题最直接、也最工程化的答案。
