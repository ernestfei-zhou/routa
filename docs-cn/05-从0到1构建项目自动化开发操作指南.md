## 1. 环境准备与安装

### 1.1 选择运行入口

Routa 提供三种运行方式，推荐使用 **Desktop 桌面版** 获得最佳体验：

| 入口 | 适用场景 | 安装方式 |
|------|----------|----------|
| **Desktop（推荐）** | 完整产品体验、可视化工作流、本地优先 | 从 [GitHub Releases](https://github.com/phodal/routa/releases) 下载安装包 |
| **Web** | 浏览器访问、自托管部署 | 从源码运行 `npm run dev` |
| **CLI** | 终端优先、脚本自动化 | `npm install -g routa-cli` |

### 1.2 Desktop 安装步骤

1. 访问 [GitHub Releases](https://github.com/phodal/routa/releases) 页面
2. 下载对应平台的安装包：
   - **macOS**: `.dmg` 文件
   - **Windows**: `.msi` 或 `.exe` 文件
   - **Linux**: `.deb` 或 `.AppImage` 文件
3. 运行安装程序，按提示完成安装
4. 启动 Routa Desktop

> **桌面版特性**：内置 Tauri + Axum 后端，自动启动 `http://127.0.0.1:3210` 后端服务，无需额外配置。

### 1.3 Web 版安装步骤（开发者自托管）

```bash
# 克隆仓库
git clone https://github.com/phodal/routa.git
cd routa

# 安装依赖
npm install

# 启动开发服务器
npm run dev
```

访问 `http://localhost:3000` 即可使用。

### 1.4 系统要求

| 项目 | 最低要求 |
|------|----------|
| 操作系统 | macOS 12+ / Windows 10+ / Ubuntu 20.04+ |
| 内存 | 8 GB RAM（推荐 16 GB） |
| 磁盘空间 | 2 GB 可用空间 |
| 网络 | 需要访问 AI 模型 API |

## 2. 首次启动与初始引导

### 2.1 启动 Routa Desktop

1. 双击桌面图标启动 Routa
2. 首次启动会自动：
   - 初始化本地 SQLite 数据库
   - 启动 Axum 后端服务（端口 3210）
   - 检测本地已安装的 ACP Agent

### 2.2 首页引导流程

首次打开 Routa 时，首页会显示 **引导卡片 (Onboarding Card)**，引导你完成三个关键步骤：

```
┌─────────────────────────────────────────────┐
│           欢迎使用 Routa                      │
│                                             │
│  ① 创建工作区     →  创建你的第一个工作区     │
│  ② 配置 Provider  →  连接 AI 模型            │
│  ③ 附加代码库     →  挂载你的项目代码         │
│                                             │
│  选择你的工作模式：                           │
│  ○ Session  ● Kanban  ○ Team                │
└─────────────────────────────────────────────┘
```

**操作步骤**：

1. 点击 **"创建工作区"** 按钮
2. 点击 **"配置 Provider"** 按钮设置模型
3. 点击 **"附加代码库"** 按钮添加代码仓库
4. 选择偏好的工作模式（Session / Kanban / Team）

### 2.3 就绪检查清单

首页底部有 **Readiness Checklist**（就绪检查清单），显示三个状态指示灯：

| 检查项 | 含义 | 颜色 |
|--------|------|------|
| **Model** | Provider 是否已配置 | 🟢 绿色 = 已就绪 / 🟡 黄色 = 未配置 |
| **Codebase** | 代码库是否已附加 | 🟢 绿色 = 已就绪 / 🟡 黄色 = 未附加 |
| **Workspace** | 工作区是否已创建 | 🟢 绿色 = 始终就绪 |

点击对应的状态按钮可直接跳转到配置页面。

## 3. 第一步：创建工作区

工作区是 Routa 中组织项目、代码库和智能体协作的核心容器。

### 3.1 通过首页创建

1. 在首页，点击引导卡片中的 **"创建工作区"** 按钮
2. 在弹出的输入框中输入工作区标题，例如 `my-todo-app`
3. 按 **Enter** 或点击 **"Create"** 按钮确认

### 3.2 通过工作区切换器创建

在应用顶部标题栏，找到 **工作区切换器**（显示当前工作区名称 + 📁 图标）：

1. 点击工作区切换器按钮
2. 在下拉菜单中点击 **"New Workspace"** 按钮
3. 输入工作区标题
4. 按 **Enter** 确认创建

### 3.3 通过侧边栏导航创建

1. 点击左侧侧边栏的 **🏠 Home** 图标返回首页
2. 使用首页的创建入口

### 3.4 切换工作区

1. 点击顶部的 **工作区切换器**
2. 在搜索框中输入关键词过滤
3. 点击目标工作区即可切换
4. 也可以直接点击列表中的工作区项

> **提示**：每个工作区是独立的，包含自己的代码库、看板、会话和团队运行记录。

## 4. 第二步：配置 Provider（模型提供者）

Provider 是 AI 模型的提供者，Routa 通过 ACP（Agent Client Protocol）协议与各 Provider 通信。

### 4.1 打开设置面板

有两种方式打开设置面板：

**方式一：通过首页就绪检查清单**
- 点击首页底部的 **Model** 状态按钮

**方式二：通过侧边栏**
- 点击左侧侧边栏底部的 **⚙️ Settings** 图标
- 或访问 `/settings` 页面

### 4.2 Provider 标签页操作

设置面板打开后，默认显示 **Providers** 标签页，包含以下区域：

#### 4.2.1 Provider Catalog（Provider 目录）

显示所有可用的 Provider，包括：

| Provider | 说明 | 状态 |
|----------|------|------|
| **Claude Code** | Anthropic Claude Code（原生 ACP 支持） | 需本地安装 claude CLI |
| **OpenCode** | OpenCode AI 编码 Agent | 需本地安装 opencode CLI |
| **Codex** | OpenAI Codex CLI（via codex-acp wrapper） | 需本地安装 codex-acp |
| **Docker OpenCode** | Docker 容器中运行的 OpenCode | 需配置 auth.json |

**操作**：
- 每个 Provider 旁边有 **复选框**，取消勾选可以隐藏不需要的 Provider
- 状态标签显示：`available`（可用）、`checking`（检测中）、`unavailable`（不可用）

#### 4.2.2 Provider Credentials（Provider 凭据）

**配置 Docker OpenCode**：
1. 找到 "Docker OpenCode" 区域
2. 在文本框中粘贴你的 `~/.local/share/opencode/auth.json` 内容
3. 格式示例：
   ```json
   {
     "zai": {
       "type": "api",
       "key": "your-api-key-here"
     }
   }
   ```
4. 点击 **"Save"** 保存

#### 4.2.3 Custom Providers（自定义 Provider）

如果你想使用自定义的 ACP Agent：
1. 点击 **"+ Add"** 按钮
2. 填写表单：
   - **Name***（必填）：Agent 名称，如 "My Agent"
   - **Command***（必填）：启动命令，如 "my-agent-cli"
   - **Args**（可选）：参数，空格分隔，如 `--acp`
   - **Description**（可选）：描述
3. 点击 **"Add"** 保存
4. 已添加的自定义 Provider 可以 **Edit** 或 **Delete**

### 4.3 Registry 标签页

切换到 **Registry** 标签页，可以：
- 浏览 ACP Agent 注册中心中的可用 Agent
- 一键安装新的 Agent
- 查看已安装 Agent 的状态

### 4.4 Roles 标签页

切换到 **Roles** 标签页，可以为不同的 Routa 角色配置默认 Provider 和模型覆盖：

| 角色 | 说明 |
|------|------|
| **ROUTA** | 主协调角色 |
| **CRAFTER** | 任务执行角色 |
| **DEVELOPER** | 开发者角色 |
| **GATE** | 质量门禁角色 |

**操作**：
1. 为每个角色选择 **Provider**（下拉选择）
2. 可选填 **Model Override**（模型覆盖），输入模型别名或模型名称
3. 选择 "Auto" 表示使用全局默认 Provider

### 4.5 Models 标签页

在 Models 标签页可以管理自定义模型定义：
- 创建模型别名（如 `fast` → `claude-3-5-haiku`）
- 配置自定义 API 端点和密钥
- 为模型设置连接参数

## 5. 第三步：附加代码库

代码库是智能体操作的代码仓库，Routa 会读取代码内容并允许智能体在其中进行修改。

### 5.1 通过首页附加

1. 在首页，点击 **Codebase** 状态按钮
2. 展开 RepoPicker 选择器

### 5.2 通过 RepoPicker 附加代码库

RepoPicker 提供三种方式添加代码库：

#### 5.2.1 从本地已有仓库选择

1. 在 RepoPicker 中，选择 **"Existing"** 标签
2. 浏览或搜索本地已有的 Git 仓库
3. 选择目标仓库
4. 选择分支（默认为当前分支）
5. 点击确认附加

#### 5.2.2 从 GitHub 克隆

1. 在 RepoPicker 中，选择 **"Clone"** 标签
2. 输入 GitHub 仓库 URL，如 `https://github.com/user/repo.git`
3. 点击 **"Clone"** 按钮
4. 等待克隆完成（进度条显示实时进度）
5. 克隆完成后自动附加到工作区

#### 5.2.3 从本地目录添加

1. 在 RepoPicker 中，选择 **"Local"** 标签
2. 输入或浏览本地目录路径
3. 选择分支
4. 确认附加

### 5.3 管理已附加的代码库

在工作区的各个页面中，你可以通过 **CodebasePicker** 组件切换当前操作的代码库：
- 如果只有一个代码库，自动选中
- 如果有多个代码库，显示下拉选择器

### 5.4 代码库自动同步

打开 Kanban 页面时，Routa 会自动同步所有附加的代码库到最新代码：
- 显示同步进度（如 "Syncing 1/3 repositories..."）
- 同步完成后状态自动消失
- 支持 `git pull` 自动拉取最新代码

## 6. 第四步：选择开发模式并开始项目

Routa 提供三种核心工作模式，首页以卡片形式展示：

```
┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│   Sessions   │  │    Kanban    │  │     Team     │
```

### 模式选择建议

| 场景 | 推荐模式 | 理由 |
|------|----------|------|
| 快速验证想法、提问 | Session | 简单直接，一问一答 |

## 7. 模式 A：Session 临时对话模式（快速验证）

Session 模式是最简单的工作模式，适合快速对话、代码问答和轻量级任务。

### 7.1 进入 Session 页面

1. 从首页点击 **"Sessions"** 卡片
2. 或点击左侧侧边栏的 **📜 Sessions** 图标
3. 路由：`/workspace/{workspaceId}/sessions`

### 7.2 Session 列表页

Session 列表页显示：

- **页面顶部**：工作区名称、Sessions 标题
- **统计信息**：总会话数、活跃会话数、最近恢复点
- **输入框**：底部的 HomeInput 输入区
- **右侧面板**（大屏）：最近会话列表

### 7.3 创建新 Session 并发送 Prompt

1. 在页面底部的输入框中输入你的需求，例如：
   ```
   帮我创建一个 React + TypeScript 的 Todo 应用项目，包含增删改查功能
   ```
2. 输入框下方的控制栏可以配置：
   - **Agent 选择**：ROUTA / CRAFTER / DEVELOPER
   - **Workspace**：当前工作区
   - **Repo/Branch**：代码库和分支选择
3. 按 **Enter** 发送
4. 系统自动创建 Session 并跳转到会话详情页

### 7.4 Session 详情页交互

会话详情页是核心的交互界面，包含以下区域：

```
┌────────────────────────────────────────────────────────────┐
│  顶部栏: Logo | 工作区 | Agent选择器 | 协议标签            │
├──────┬─────────────────────────────────────┬───────────────┤
│ 左侧 │           中间聊天区域               │   右侧面板    │
│ 侧栏 │                                     │  (可调整宽度) │
│      │  ┌─────────────────────────────┐    │              │
│ 会话 │  │ User: 帮我创建 Todo 应用     │    │  Task Panel  │
│ 列表 │  │                             │    │  CRAFTERs    │
│      │  │ Assistant: 好的，我来帮你...  │    │  视图        │
│ 技能 │  │ [创建文件] [编辑代码]...     │    │              │
│      │  └─────────────────────────────┘    │              │
│      │                                     │              │
│      │  ┌─────────────────────────────┐    │              │
│      │  │ 输入框 (TiptapInput)         │    │              │
│      │  └─────────────────────────────┘    │              │
├──────┴─────────────────────────────────────┴───────────────┤
└────────────────────────────────────────────────────────────┘
```

**各区域功能**：

| 区域 | 功能 | 操作 |
|------|------|------|
| **左侧侧栏** | 会话列表、技能列表 | 点击切换会话，搜索技能 |
| **中间聊天区** | 对话消息、工具调用、代码变更 | 滚动查看历史，回复交互 |
| **右侧面板** | 任务面板、CRAFTERs 视图 | 拖拽调整宽度 |
| **顶部栏** | Agent 选择、Provider 切换 | 切换角色和模型 |
| **输入框** | TiptapInput 富文本输入 | 支持 `/` 触发技能、`@` 文件引用 |

### 7.5 在 Session 中切换 Agent

在顶部栏的 Agent 选择器中：
1. 点击当前 Agent 名称旁的下拉箭头
2. 选择新的 Agent 角色（ROUTA / CRAFTER / DEVELOPER / GATE）
3. 也可以选择 Specialist 专家角色

### 7.6 回复智能体提问

当智能体需要用户确认时（AskUserQuestion），会在聊天区域显示交互式表单：
1. 阅读智能体的问题
2. 在输入框中填写回复
3. 点击 **"Submit"** 提交

### 7.7 恢复之前的 Session

1. 在 Session 列表页，右侧面板显示最近的会话
2. 点击任意会话即可进入详情页继续对话
3. 系统会自动恢复会话上下文

## 8. 模式 B：Kanban 看板自动化模式（项目开发）

Kanban 模式是 Routa 的核心工作模式，支持列自动化、Git 集成和任务驱动的开发流程。

### 8.1 进入 Kanban 页面

1. 从首页点击 **"Kanban"** 卡片
2. 或点击左侧侧边栏的 **📊 Kanban** 图标
3. 路由：`/workspace/{workspaceId}/kanban`

> **注意**：访问工作区根路由 `/workspace/{workspaceId}` 会自动重定向到 Kanban 页面。

### 8.2 Kanban 页面结构

```
┌──────────────────────────────────────────────────────────────┐
│  顶部: 工作区切换器 | 看板标题 | 操作按钮                      │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐    │
│  │  Backlog │  │  To Do   │  │ In Progress│ │  Done    │    │
│  │          │  │          │  │           │  │          │    │
│  │ ┌──────┐ │  │ ┌──────┐ │  │ ┌──────┐  │  │ ┌──────┐ │    │
│  │ │Card 1│ │  │ │Card 2│ │  │ │Card 3│  │  │ │Card 4│ │    │
│  │ └──────┘ │  │ └──────┘ │  │ └──────┘  │  │ └──────┘ │    │
│  │ ┌──────┐ │  │          │  │           │  │          │    │
│  │ │Card 5│ │  │          │  │           │  │          │    │
│  │ └──────┘ │  │          │  │           │  │          │    │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘    │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 8.3 创建看板

1. 在 Kanban 页面，点击 **"New Board"** 按钮
2. 在创建看板弹窗中配置：
   - **Board Title**（看板标题）：如 "Todo App Development"
   - **Columns**（列）：可以自定义列名和顺序
   - **Auto Provider**：选择自动化使用的 Provider
3. 点击 **"Create"** 确认

### 8.4 创建任务卡片

1. 在看板的某列中，点击 **"+"** 按钮或 **"Add Card"**
2. 在弹窗中填写：
   - **Title**（标题）：任务标题
   - **Description**（描述）：详细需求
   - **Assignee**（分配）：选择 Specialist
   - **Priority**（优先级）：High / Medium / Low
   - **Labels**（标签）：自定义标签
3. 点击 **"Create"** 创建卡片

### 8.5 配置列自动化

列自动化是 Kanban 模式的核心功能，当卡片移动到某列时自动触发智能体：

   - **Specialist**：选择专家角色
   - **Steps**：配置自动化步骤
3. 保存配置

**自动化工作流**：
- 卡片移入列 → 自动创建 Session → 智能体按配置执行任务 → 完成后更新卡片状态

### 8.6 拖拽移动卡片

2. 可以查看：
   - 任务描述和进度
   - 关联的 Session 链接
   - Git 变更状态
   - Fitness 检查结果
3. 可以手动触发 Agent 处理

### 8.8 Git 变更监控

Kanban 页面会实时显示代码库的 Git 变更：
- 每个代码库的修改文件数

## 12. 配置 Schedule 定时调度

Schedule 允许你创建定时触发的自动化任务，类似 cron job。

### 12.1 进入 Schedule 设置页

1. 点击左侧侧边栏的 **⚙️ Settings** 图标
2. 在设置中心导航中，点击 **"Schedules"**
3. 路由：`/settings/schedules`

### 12.2 Schedule 页面结构

- 顶部显示工作区切换器（Schedule 是工作区级别的配置）
- 显示 Tick 端点信息：`/api/schedules/tick`
- Schedule 列表和管理面板

### 12.3 创建 Schedule

1. 在 Schedule 面板中，点击 **"Add Schedule"**
2. 配置调度规则：
   - **Name**：调度名称
   - **Cron Expression**：cron 表达式（如 `0 9 * * 1-5` 表示工作日每天9点）
   - **Action**：触发动作
   - **Workspace**：关联工作区
   - **Provider**：使用的 Provider
   - **Prompt**：触发的提示词
3. 保存配置

### 12.4 Schedule 工作原理

外部 cron 服务（如 GitHub Actions）定期调用 `/api/schedules/tick` 端点，Routa 根据配置的 cron 表达式判断是否需要触发任务。

## 14. 配置 Webhook 自动触发

Webhook 允许外部事件（如 GitHub 事件）自动触发智能体执行。

### 14.1 进入 Webhook 设置页

1. 点击左侧侧边栏的 **⚙️ Settings** 图标
2. 在设置中心导航中，点击 **"Webhooks"**
3. 路由：`/settings/webhooks`

### 14.2 Webhook URL

Routa 提供统一的 Webhook 接收端点：

```
{your-domain}/api/webhooks/github
```

### 14.3 配置 GitHub Webhook

1. 在 GitHub 仓库的 Settings → Webhooks 中添加新 Webhook
2. **Payload URL**：填入 Routa 的 Webhook URL
3. **Content type**：选择 `application/json`
4. **Secret**：设置签名密钥（需与 Routa 配置一致）
5. **Events**：选择需要监听的事件类型

### 14.4 在 Routa 中配置 Webhook 触发

1. 在 Webhook 设置页，点击 **"Add Trigger"**
2. 配置：
   - **Event Type**：事件类型（issues, pull_request, push 等）
   - **Event Action**：动作（opened, closed, merged 等）
   - **Provider**：触发的 Provider
   - **Prompt Template**：提示词模板（支持变量替换）
3. 保存配置

## 17. 完整实战：从0到1构建一个 Todo 应用

本节通过一个完整的实战示例，展示如何使用 Routa 从零开始构建一个 Todo 应用。

### 17.1 步骤一：创建工作区

1. 启动 Routa Desktop
2. 在首页点击 **"创建工作区"**
3. 输入标题 `todo-app-project`
4. 点击 **"Create"** 创建

### 17.2 步骤二：配置 Provider

1. 点击首页底部的 **Model** 状态按钮
2. 在 Provider Catalog 中确认至少一个 Provider 显示为 `available`
3. 如果使用 Docker OpenCode，在 Credentials 区域配置 auth.json
4. 关闭设置面板

### 17.3 步骤三：克隆/创建代码库

1. 点击首页底部的 **Codebase** 状态按钮
2. 在 RepoPicker 中选择 **"Clone"** 标签
3. 输入你的空仓库 URL（或选择 Local 创建新目录）
4. 等待克隆完成

### 17.4 步骤四：选择 Kanban 模式开始开发

1. 在首页点击 **"Kanban"** 卡片
2. 进入 Kanban 页面

### 17.5 步骤五：创建看板

1. 点击 **"New Board"**
2. 标题设为 "Todo App Development"
3. 创建以下列：
   - **Backlog**：待规划的需求
   - **To Do**：已规划待开发
   - **In Progress**：开发中
   - **Review**：代码审查
   - **Done**：已完成
4. 点击 **"Create"**

### 17.6 步骤六：创建任务卡片

在 **Backlog** 列中创建以下卡片：

| 卡片标题 | 描述 | 优先级 |
|----------|------|--------|
| 项目初始化 | 使用 Vite + React + TypeScript 初始化项目 | High |
| 数据模型设计 | 设计 Todo 数据模型和 API 接口 | High |
| 前端界面开发 | 实现 Todo 列表、添加、删除、编辑界面 | Medium |
| 后端 API 开发 | 实现增删改查 REST API | Medium |
| 添加样式 | 使用 Tailwind CSS 美化界面 | Low |
| 编写测试 | 添加单元测试和集成测试 | Medium |

### 17.7 步骤七：配置列自动化

为 **In Progress** 列配置自动化：
1. 点击列标题的 ⚙️ 图标
2. 开启自动化
3. 选择 Provider：Claude Code（或你的首选 Provider）
4. 选择 Role：DEVELOPER
5. 保存

### 17.8 步骤八：开始开发

1. 将 **"项目初始化"** 卡片拖拽到 **"In Progress"** 列
2. 自动触发智能体，开始执行项目初始化
3. 智能体会：
   - 创建 Vite + React + TypeScript 项目
   - 安装必要依赖
   - 配置项目结构
4. 完成后卡片自动更新状态

### 17.9 步骤九：逐步推进

依次将任务卡片从 Backlog → To Do → In Progress：
1. **数据模型设计**：智能体创建类型定义和 API 接口
2. **后端 API 开发**：智能体实现 REST API
3. **前端界面开发**：智能体实现 React 组件
4. **添加样式**：智能体应用 Tailwind CSS
5. **编写测试**：智能体添加测试用例

### 17.10 步骤十：使用 Team 模式（可选进阶）

如果你想多个智能体并行开发：

1. 切换到 **Team** 页面
2. 输入需求：
   ```
   并行开发 Todo 应用：前端使用 React + TypeScript + Tailwind，后端使用 Express + SQLite，包含完整的增删改查功能和测试
   ```
3. Team Lead 会自动：
   - 分解需求
   - 委派 Frontend Dev 开发前端
   - 委派 Backend Dev 开发后端
   - 委派 QA 编写测试
   - 协调各成员进度

### 17.11 步骤十一：代码审查

1. 在 Kanban 中将卡片移到 **Review** 列
2. 配置 Review 列自动化使用 Code Reviewer Specialist
3. 智能体自动审查代码并给出反馈

### 17.12 步骤十二：完成项目

所有卡片移到 **Done** 列后，项目开发完成！

## 20. 常见问题与故障排查

### 20.1 Provider 显示 "unavailable"

**原因**：对应的 CLI 工具未安装或不在 PATH 中。

**解决**：
1. 确认 CLI 工具已安装（如 `claude`、`opencode`）
2. 在终端中验证：`which claude` 或 `claude --version`
3. 重启 Routa Desktop 让它重新检测

### 20.2 Docker OpenCode 启动失败

**原因**：未配置 auth.json 或配置格式错误。

**解决**：
1. 打开设置面板 → Providers → Provider Credentials
2. 粘贴正确的 auth.json 内容
3. 确保 JSON 格式正确
4. 点击 Save 后重试

### 20.3 代码库附加失败

**原因**：路径不正确或不是 Git 仓库。

**解决**：
1. 确认路径是有效的 Git 仓库
2. 在终端中 `git status` 验证仓库状态
3. 使用 Clone 方式从 GitHub 克隆

### 20.4 Session 创建后无响应

**原因**：Provider 连接问题或认证失败。

**解决**：
1. 检查 Provider 状态是否为 `available`
2. 查看是否弹出认证错误提示
3. 尝试切换到其他 Provider
4. 查看后台日志（开发模式）

### 20.5 Kanban 自动化未触发

**原因**：列自动化未启用或配置不正确。

**解决**：
1. 确认列的自动化开关已开启
2. 检查 Provider 和 Role 配置
3. 确认代码库已附加
4. 查看 Messages 页面中的后台任务状态

### 20.6 Team Run 卡住不动

**原因**：Team Lead 无法委派任务或子 Session 创建失败。

**解决**：
1. 检查 Specialist 配置是否正确
2. 确认所有团队成员角色都有对应的 Specialist
3. 查看 Trace 页面了解执行细节
4. 尝试重新发起 Team Run

### 20.7 页面加载缓慢

**解决**：
1. 检查网络连接（需要访问 AI API）
2. 清除浏览器缓存
3. 重启 Routa Desktop
4. 检查后端服务状态（`http://127.0.0.1:3210`）

### 20.8 桌面版后端服务未启动

**解决**：
1. 检查 3210 端口是否被占用
2. 在终端中运行：`lsof -i :3210`（macOS/Linux）或 `netstat -ano | findstr :3210`（Windows）
3. 如果被占用，结束占用进程或修改 Routa 配置
4. 重启 Routa Desktop

## 附录 D：ACP 协议与 Provider 架构

Routa 使用 ACP（Agent Client Protocol）与各 AI Provider 通信。

### Provider 检测流程

1. **本地检测**：检查本地是否安装了对应的 CLI 工具
2. **注册中心查询**：从 ACP Registry 获取可用的 Provider
3. **状态刷新**：定期检测 Provider 的可用性

### Provider 优先级

1. 用户手动选择的 Provider
2. 上次使用的 Provider（持久化到 localStorage）
3. 第一个 `available` 状态的 Provider

### 自定义 Provider

支持添加任意 ACP 兼容的 Agent：
- 自定义启动命令和参数
- 独立的 Provider ID
- 可在 Roles 配置中指定使用
