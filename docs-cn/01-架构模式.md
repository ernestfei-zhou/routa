# Routa.js 架构模式详解

> 版本：0.17.0 | 作者：Phodal Huang | 许可证：MIT

## 一、项目定位

Routa.js 是一个**工作区优先的多智能体协调平台**（Workspace-First Multi-Agent Coordination Platform），旨在为 AI 开发工作流提供统一的任务编排、Agent 管理、代码协作和项目治理能力。

## 二、双栈架构（Dual-Backend Architecture）

Routa.js 采用 **TypeScript (Next.js) + Rust (Axum)** 双栈架构，两者并非两个独立产品，而是同一产品在不同部署模型下的两种运行时表面：

| 维度 | Web 运行时 | Desktop 运行时 |
|------|-----------|---------------|
| **前端** | Next.js App Router | Tauri + Next.js 静态导出 |
| **后端** | Next.js API Routes | Axum HTTP 服务器 |
| **核心逻辑** | `src/core/` (TypeScript) | `crates/routa-core/` (Rust) |
| **持久化** | Postgres (Neon) / SQLite / 内存 | SQLite + 文件系统 |
| **部署** | Vercel / 自托管 | 桌面应用 (Tauri) |
| **端口** | 3000 | 3210 |

### 核心原则：双后端语义一致性

Web 和 Desktop 的领域语义、API 形状和 Agent 协调行为必须保持一致，由 `api-contract.yaml` 作为契约保证。两者差异仅体现在部署模型和存储方式上。

## 三、分层架构

双运行时遵循相同的分层架构模型：

```text
┌─────────────────────────────────────────────────────┐
│  表现层 (Presentation)                                │
│  React 页面、工作区视图、会话详情、看板、设置、追踪     │
├─────────────────────────────────────────────────────┤
│  API / 传输层 (API / Transport)                      │
│  Next.js Route Handlers 或 Axum Routers              │
├─────────────────────────────────────────────────────┤
│  协议适配层 (Protocol Adapters)                       │
│  REST / MCP / ACP / A2A / AG-UI / SSE / JSON-RPC    │
├─────────────────────────────────────────────────────┤
│  领域服务层 (Domain Services)                         │
│  编排、看板自动化、工作流执行、笔记、审查、调度、       │
│  追踪、Harness、共享会话、Worker 派发                  │
├─────────────────────────────────────────────────────┤
│  存储 / 注册层 (Stores / Registries)                  │
│  工作区、任务、会话、笔记、代码库、工作树、调度、       │
│  工件、技能                                           │
├─────────────────────────────────────────────────────┤
│  持久化 / 运行时层 (Persistence / Runtime)            │
│  Postgres、SQLite、内存、JSONL 追踪、本地进程、        │
│  Docker、文件系统                                     │
└─────────────────────────────────────────────────────┘
```

**依赖方向**：必须自上而下。UI 和传输层依赖领域服务；存储和运行时层不应依赖 UI 关注点。

## 四、仓库结构

### TypeScript 层 (`src/`)

| 目录 | 职责 |
|------|------|
| `src/app/` | Next.js App Router 页面和 API 路由 |
| `src/client/` | 客户端组件、Hooks、视图模型、A2UI 辅助 |
| `src/core/` | TypeScript 领域逻辑：Store、ACP/MCP、看板自动化、工作流、笔记、工具 |

### Rust 层 (`crates/`)

| Crate | 职责 |
|-------|------|
| `routa-core` | 传输无关的核心业务逻辑：数据模型、Store、EventBus、编排引擎、工作流引擎、MCP、沙箱、技能、追踪 |
| `routa-server` | 基于 Axum 的 HTTP 服务器，包含所有 API 路由处理 |
| `routa-cli` | CLI 入口和 ACP 服务命令 |
| `routa-rpc` | 传输无关的 JSON-RPC 2.0 分发器（HTTP/Tauri IPC/CLI/JS bindgen） |
| `routa-scanner` | 代码库扫描工具 |
| `trace-parser` | Agent 追踪记录解析器 |
| `feature-trace` | 特性追踪 |
| `harness-monitor` | 项目表面信号监控（四层 Harness 循环：Context → Run → Observe → Govern） |
| `entrix` | 质量门禁运行时和 CLI |

### 其他关键目录

| 目录 | 职责 |
|------|------|
| `apps/desktop/` | Tauri 桌面应用封装 |
| `resources/` | 预定义工作流 YAML、专家角色配置、Canvas SDK、审查规则 |
| `docs/` | 架构文档、ADR、设计文档、实施计划、Fitness 规范 |
| `drizzle/` / `drizzle-sqlite/` | Postgres / SQLite 数据库迁移 |
| `e2e/` | Playwright 端到端测试 |

## 五、系统工厂与共享状态

### TypeScript `RoutaSystem`

`src/core/routa-system.ts` 是 Next.js 运行时的**中央装配点**，负责：

- 组装所有 Store（Agent、Conversation、Task、Note、Workspace、Codebase、Worktree、BackgroundTask、Schedule、KanbanBoard、Artifact、WorkflowRun）
- 创建 `EventBus` 进行进程内协调
- 注册 MCP 工具表面（`AgentTools`、`NoteTools`、`WorkspaceTools`、`KanbanTools`）
- 配置 CRDT 文档管理和笔记广播
- 配置权限存储
- 启动看板工作流编排器

**三种存储模式**：
1. **InMemory** — 无数据库，用于快速开发/测试
2. **Postgres** — `DATABASE_URL` 设置时，通过 Neon Serverless + Drizzle ORM（Web/Vercel）
3. **SQLite** — `ROUTA_DB_DRIVER=sqlite` 或本地 Node 开发时，通过 better-sqlite3

通过 `getDatabaseDriver()` 自动选择，单例挂载在 `globalThis` 上以在 Next.js HMR 中存活。

### Rust `AppState`

`crates/routa-core/src/state.rs` 在 Axum 服务器中扮演相同角色：

- 核心存储：Workspace、Codebase、Worktree、Agent、Artifact、Task、Kanban、Note、Schedule、Conversation、AcpSession
- `AcpManager`：ACP 进程生命周期管理
- `AcpBinaryManager` / `AcpInstallationState` / `AcpRuntimeManager` / `AcpWarmupService`：二进制管理、安装状态、运行时管理、热身服务
- `SkillRegistry`：技能发现与注册
- `EventBus`：事件总线
- `SandboxManager`：Docker 沙箱管理
- `DockerState`：Docker 检测和进程管理

## 六、协议栈

| 协议 | 主要端点 | 角色 |
|------|---------|------|
| **REST** | `/api/*` | CRUD 和产品面向操作 |
| **MCP** | `/api/mcp`, `/api/mcp/tools` | 工具执行和协作 Agent 能力 |
| **ACP** | `/api/acp` 及相关路由 | Agent 进程的创建、提示、流式、安装、热身和管理 |
| **A2A** | `/api/a2a/*` | Agent 间互操作协议 |
| **AG-UI** | `/api/ag-ui` | UI 面向的 Agent 流协议 |
| **A2UI** | `/api/a2ui/*` | 仪表盘导向的 UI 协议 |
| **SSE** | ACP、笔记、AG-UI 等端点 | 增量更新推送到前端 |

## 七、实时与事件机制

两层实时机制：

1. **传输层流式**：主要使用 SSE 进行会话、笔记和协议更新
2. **进程内事件**：TypeScript 和 Rust 运行时各自的 `EventBus`

支持的场景：
- Agent 生命周期追踪
- 看板自动运行队列消费
- 笔记变更传播
- 工作流和后台任务协调
- 会话和追踪表面的 UI 刷新触发

## 八、持久化模型

### Web 端
- 主要持久化目标：Postgres（`DATABASE_URL` 配置时）
- SQLite 支持本地 Node 开发
- 内存模式用于测试和轻量级运行时场景

### Desktop 端
- SQLite 是正常持久化存储
- 文件系统状态也是持久化的一部分：会话 JSONL 追踪、仓库、工作树、Agent 二进制、本地配置

### 追踪与历史
- 会话和追踪历史可存储在数据库记录、JSONL 文件或两者兼有（取决于运行时）
- 追踪数据是一等调试和归因机制，不是附属日志流

## 十、当前过渡区域

项目正在完成工作区为中心的归一化，当前存在的过渡性问题：

- 部分路径在缺少 `workspaceId` 时仍回退到 `"default"`
- 部分引导/运行时流程仍假设默认工作区存在
- 并非每个持久化实现在 TypeScript 和 Rust 间完全对称
- 部分工作流运行持久化仍为内存模式

**`"default"` 应视为过渡脚手架，不是目标领域模型。**

## 十一、设计哲学

- **领域对象优先**：先思考领域对象，而非辅助函数
- **薄壳 + 厚核心**：CLI/API/UI 主要做派发，真正逻辑在 core/usecases
- **显式优于隐式**：优化显式性、可读性和交付价值
- **协议导向编排**：通过协议层（REST/MCP/ACP/A2A）标准化集成，而非 Provider 特定耦合
- **本地优先执行**：Desktop 模式倾向 SQLite、本地 Agent 二进制、本地工作树和追踪文件
- **Provider 抽象**：不同 Agent CLI 和运行时通过适配器层归一化，不泄露 Provider 特定协议细节到系统中
