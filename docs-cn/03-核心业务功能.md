# Routa.js 核心业务功能详解

> 版本：0.17.0 | 本文档详细列出 Routa.js 平台的所有核心业务功能模块、领域实体、API 表面和预置资源

---

## 一、领域实体与关系模型

### 1.1 核心实体关系图

```text
Workspace（工作区 — 顶级组织单元）
  ├── Codebase（代码库 — 多个/工作区）
  │     ├── Worktree（工作树 — 并行开发分支）
  │     ├── RepoWiki（仓库知识库）
  │     └── RepoSlide（仓库演示文稿）
  ├── Agent（智能体）
  │     ├── Role: ROUTA / CRAFTER / GATE / DEVELOPER
  │     ├── ModelTier: SMART / BALANCED / FAST
  │     ├── Status: PENDING → ACTIVE → COMPLETED / ERROR / CANCELLED
  │     └── parent_id → 可组成父子层级
  ├── Task（任务）
  │     ├── TaskLaneSession（车道会话 — 多步执行追踪）
  │     ├── Status: TODO / IN_PROGRESS / IN_REVIEW / DONE / CANCELLED
  │     ├── Priority: LOW / MEDIUM / HIGH / CRITICAL
  │     ├── 依赖关系（dependencies）
  │     └── Artifact 附件
  ├── Note（笔记 — CRDT 协同编辑）
  │     └── Type: Spec / Task / General
  ├── KanbanBoard（看板）
  │     ├── KanbanColumn（列 — 含自动化步骤）
  │     ├── KanbanCard（卡片 — 关联 Task）
  │     └── KanbanConfig（看板配置 — 交付策略/列策略）
  ├── Artifact（工件 — Agent 间通信）
  │     └── Type: Screenshot / TestResults / CodeDiff / Logs / Canvas
  ├── Schedule（定时调度 — Cron 表达式）
  ├── AcpSession（ACP 会话 — Agent 进程交互）
  ├── Conversation（对话）
  ├── CustomMcpServer（自定义 MCP 服务器）
  ├── Skill（技能 — SKILL.md）
  ├── Specialist（专家 — 预定义角色配置）
  ├── BackgroundTask（后台任务 — 异步工作）
  ├── CanvasArtifact（画布工件 — 可视化仪表盘）
  ├── Memory（记忆 — 工作区上下文记录）
  └── Message（消息 — 通信记录）
```

### 1.2 核心实体字段详情

#### Agent（智能体）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | String | 唯一标识 |
| `name` | String | Agent 名称 |
| `role` | AgentRole | 角色：ROUTA(协调) / CRAFTER(工匠) / GATE(守门) / DEVELOPER(开发) |
| `model_tier` | ModelTier | 模型层级：SMART(强) / BALANCED(均衡) / FAST(快) |
| `workspace_id` | String | 所属工作区 |
| `parent_id` | Option\<String\> | 父 Agent ID（子 Agent 时有值） |
| `status` | AgentStatus | 状态：PENDING → ACTIVE → COMPLETED / ERROR / CANCELLED |
| `metadata` | HashMap | 扩展元数据 |

#### Task（任务）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | String | 唯一标识 |
| `title` | String | 任务标题 |
| `description` | Option\<String\> | 任务描述 |
| `status` | TaskStatus | TODO / IN_PROGRESS / IN_REVIEW / DONE / CANCELLED |
| `priority` | TaskPriority | LOW / MEDIUM / HIGH / CRITICAL |
| `workspace_id` | String | 所属工作区 |
| `assignee_id` | Option\<String\> | 指派 Agent |
| `parent_task_id` | Option\<String\> | 父任务（子任务分解） |
| `dependencies` | Vec\<String\> | 依赖的任务 ID 列表 |
| `lane_session_id` | Option\<String\> | 当前车道会话 ID |
| `specialist` | Option\<String\> | 关联专家角色 |
| `required_artifacts` | Vec\<String\> | 完成所需的工件类型 |
| `evidence` | Vec\<TaskEvidence\> | 任务执行证据 |

#### KanbanBoard（看板）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | String | 唯一标识 |
| `title` | String | 看板标题 |
| `workspace_id` | String | 所属工作区 |
| `columns` | Vec\<KanbanColumn\> | 列定义（含自动化配置） |
| `cards` | Vec\<KanbanCard\> | 卡片列表 |
| `config` | KanbanConfig | 看板配置（交付策略、列转移策略） |

#### Artifact（工件）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | String | 唯一标识 |
| `task_id` | String | 关联任务 |
| `artifact_type` | ArtifactType | Screenshot / TestResults / CodeDiff / Logs / Canvas |
| `content` | String | 工件内容 |
| `requested_by` | Option\<String\> | 请求方 Agent |
| `provided_by` | Option\<String\> | 提供方 Agent |

#### CanvasArtifact（画布工件）

| 字段 | 类型 | 说明 |
|------|------|------|
| `metadata` | CanvasArtifactMetadata | 元数据 |
| `source` | Option\<String\> | 源代码 |
| `data` | Option\<JsonValue\> | 数据载荷 |

`CanvasArtifactMetadata` 包含：`render_mode`（dynamic/prebuilt）、`canvas_type`（fitness_overview）、`title`、`schema_version`、`workspace_id`、`codebase_id`、`repo_path`

### 3.2 MCP Rust 服务器

`crates/routa-core/src/mcp/mod.rs` 通过 rmcp SDK 将 Routa 的工具暴露给 AI 助手：

- **Agent 工具**：`list_agents` — 列出工作区所有 Agent
- **Task 工具**：`list_tasks`、`find_ready_tasks` — 列出任务、查找可执行任务
- **Note 工具**：`list_notes`、`read_spec` — 读取笔记和规范
- **Workspace 工具**：`list_workspaces` — 列出工作区
- **Skill 工具**：`list_skills` — 列出已发现技能

### 3.3 JSON-RPC 分发器

`crates/routa-core/src/rpc/` 实现传输无关的 JSON-RPC 2.0 路由：

- 支持从 HTTP、Tauri IPC、CLI、JS bindgen 调用
- 方法命名空间：`agents`、`tasks`、`notes`、`workspaces`、`skills`、`kanban`
- Kanban 子方法：`automation`、`boards`、`cards`、`handoffs`、`queries`

### 6.4 API 路径解析

前端和桌面 API 调用使用统一的路径组装：

- `resolveApiPath`（`src/client/config/backend.ts`）：统一补全 `/api` 前缀，必要时拼接后端 base URL
- `desktopAwareFetch`（`src/client/utils/diagnostics.ts`）：Tauri 桌面静态运行时自动回退到 `http://127.0.0.1:3210`
- **禁止**在前端/桌面再次直接写 `fetch('/api/...')`

## 七、数据存储架构

### 7.1 Store 接口层级

```text
┌──────────────────────────────────────────┐
│           Store 接口层                    │
│  AgentStore / TaskStore / NoteStore ...  │
├──────────────────────────────────────────┤
│        具体实现层（按存储后端选择）         │
│  InMemory*Store / Pg*Store / Sqlite*Store│
├──────────────────────────────────────────┤
│           数据库层                        │
│  内存对象 / Postgres (Drizzle) / SQLite   │
└──────────────────────────────────────────┘
```

### 7.2 数据库 Schema

TypeScript 层定义了两套 Schema：
- `src/core/db/schema.ts`（Postgres，29.66KB）
- `src/core/db/sqlite-schema.ts`（SQLite，28.03KB）

Drizzle ORM 管理迁移：
- `drizzle/` — Postgres 迁移文件
- `drizzle-sqlite/` — SQLite 迁移文件
