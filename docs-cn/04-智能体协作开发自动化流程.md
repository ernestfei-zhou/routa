## 1. 环境准备与安装

### 1.1 选择入口

Routa 提供三种运行入口：

| 入口 | 适用场景 | 安装方式 |
|------|----------|----------|
| **Desktop** | 完整产品体验、可视化工作流、本地优先 | 从 [GitHub Releases](https://github.com/phodal/routa/releases) 下载 |
| **CLI** | 终端优先、脚本自动化 | `npm install -g routa-cli` 或 `cargo install routa-cli` |
| **Web** | 浏览器访问、自托管部署 | 从源码运行 |

### 1.2 Desktop 安装

1. 从 [GitHub Releases](https://github.com/phodal/routa/releases) 下载对应平台安装包
2. 安装并启动 Routa Desktop
3. 桌面版内置 Tauri + Axum 后端，端口默认 `http://127.0.0.1:3210`

### 1.3 CLI 安装

```bash
# npm 安装（推荐）
npm install -g routa-cli

# Cargo 安装（Rust 优先用户）
cargo install routa-cli

# 验证安装
routa --help
routa --version
```

### 1.4 Web 从源码运行

```bash
# 克隆仓库
git clone https://github.com/phodal/routa.git
cd routa

# 安装依赖
npm install --legacy-peer-deps

# 启动开发服务器
npm run dev
# 打开 http://localhost:3000

# 如果需要连接桌面后端
ROUTA_RUST_BACKEND_URL="http://127.0.0.1:3210" npm run dev
```

### 1.5 Docker 部署

```bash
# SQLite 模式
docker compose up --build

# PostgreSQL 模式
docker compose --profile postgres up --build
```

## 2. 核心概念总览

### 2.1 实体关系图

```
Workspace（工作区，顶级范围）
  ├── Codebases（代码库）
  │     └── Worktrees（工作树/执行副本）
  ├── Kanban Boards（看板）
  │     └── Columns（泳道） + Tasks（卡片）
  ├── Sessions（ACP 会话）
  ├── Tasks（任务）
  ├── Notes（笔记，CRDT 实时协作）
  ├── Schedules（定时调度）
  ├── Artifacts（智能体间工件）
  ├── Skills（技能）
  └── Memories（上下文记忆）
```

### 2.2 核心不变量

- 所有用户可见的工作区资源必须有显式的工作区范围（ADR 0003）
- 代码库属于一个工作区
- 会话和看板界面反映当前选中的工作区
- 看板不仅仅是 UI——修改看板列状态可能触发自动化（ADR 0004）

### 2.3 三层角色体系

| 层级 | 角色 | 职责 | 关键规则 |
|------|------|------|----------|
| **核心角色** | ROUTA（协调者） | 规划、委派、验证 | **绝不直接编辑文件**，通过 `delegate_task_to_agent` 委派 |
| | CRAFTER（实现者） | 实现代码 | 保持任务范围，不扩展范围，完成后 `report_to_parent` |
| | GATE（验证者） | 仅对照验收标准验证 | 证据驱动，不允许部分批准，输出明确裁决 |
| **看板泳道专家** | Backlog Refiner | 澄清需求，输出 canonical YAML story block | 不实现代码 |
| | Todo Orchestrator | 不信任上游输出，重新解析 YAML，追加执行简报 | 拒绝弱 story |
| | Dev Crafter | 不信任计划，验证可执行后才编码 | 实现+提交+记录 Dev Evidence |
| | Review Guard | 独立验证每个 AC，要求测试和干净 git 状态 | 拒绝则回退到 Dev |
| | Done Reporter | 追加完成摘要 | 不可再前进 |
| | Blocked Resolver | 分类阻塞，路由回正确泳道 | 有具体下一步才路由 |
| **团队角色** | Agent Lead | 团队协调 | 委派子任务 |
| | Frontend Dev | 前端实现 | 不碰后端代码 |
| | Backend Dev | 后端实现 | 不碰前端代码 |
| | QA | 测试 | 验证 AC |
| | Researcher | 调研 | 不实现代码 |
| | UX Designer | 设计 | 输出设计规范 |

## 3. 第一步：创建工作区

工作区是所有实体的顶级协调边界。每个会话、任务、笔记、看板、代码库都归属于一个工作区。

### 3.1 通过 UI

1. 启动 Routa 后，首页显示 `WorkspaceSwitcher` 组件
2. 点击「创建工作区」，输入名称
3. 创建后自动导航到 `/workspace/[workspaceId]`

### 3.2 通过 API

```bash
# 创建工作区
curl -X POST http://localhost:3000/api/workspaces \
  -H "Content-Type: application/json" \
  -d '{"name": "我的项目", "description": "项目描述"}'

# 列出所有工作区
curl http://localhost:3000/api/workspaces
```

### 3.3 通过 CLI

```bash
routa workspace list
```

## 4. 第二步：配置 Provider（模型提供者）

至少启用一个 Provider 才能让智能体运行。

### 4.1 支持的 Provider

| Provider | 适配器 | 说明 |
|----------|--------|------|
| Claude Code SDK | `claude-code-sdk` | Anthropic Claude，流式 JSON 协议 |
| OpenCode | `opencode-sdk` | ACP 原生协议 |
| Docker OpenCode | `docker-opencode` | Docker 容器中运行的 OpenCode |
| Workspace Agent | `workspace-agent` | 内置轻量级智能体 |

### 4.2 通过 UI 配置

1. 首页 → Settings 面板
2. 选择 Provider 类型，填入 API Key 和 Base URL
3. 点击保存

### 4.3 通过 API 配置

```bash
# 查看 ACP 注册表（可用运行时）
curl http://localhost:3000/api/acp/registry

# 安装运行时
curl -X POST http://localhost:3000/api/acp/install \
  -H "Content-Type: application/json" \
  -d '{"runtimeId": "claude-code-sdk"}'

# 预热运行时
curl -X POST http://localhost:3000/api/acp/warmup \
  -H "Content-Type: application/json" \
  -d '{"runtimeId": "claude-code-sdk"}'
```

### 4.4 关键环境变量

```bash
# 模型配置

ANTHROPIC_BASE_URL=https://api.anthropic.com
```

## 6. 模式 A：Session 临时对话模式

Session 模式适合临时对话、快速问答和探索性工作。

### 6.1 操作流程

```
创建会话 → 发送提示词 → 接收流式响应 → 查看追踪 → 继续对话或结束
```

### 6.2 通过 UI

1. 在工作区页面 → Sessions 面板
2. 点击「新建会话」
3. 选择 Provider、Role（如 ROUTA、CRAFTER、GATE）
4. 输入提示词，开始对话
5. 路由：`/workspace/[workspaceId]/sessions/[sessionId]`

### 6.3 通过 API（ACP 协议）

ACP（Agent Client Protocol）是 Routa 的核心执行协议，所有智能体运行时通过 ACP 适配器归一化。

```bash
# 1. initialize — 协商协议版本和能力
curl -X POST http://localhost:3000/api/acp \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "id": "1",
    "method": "initialize",
    "params": {
      "protocolVersion": "0.1"
    }
  }'

# 2. session/new — 创建新会话
curl -X POST http://localhost:3000/api/acp \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "id": "2",
    "method": "session/new",
    "params": {
      "cwd": "/path/to/repo",
      "provider": "claude-code-sdk",
      "role": "ROUTA",
      "toolMode": "full",
      "workspaceId": "your-workspace-id",
      "name": "架构分析会话"
    }
  }'
# 返回：{ "result": { "sessionId": "abc-123" } }

# 3. session/prompt — 发送提示词
curl -X POST http://localhost:3000/api/acp \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc": "2.0",
    "id": "3",
    "method": "session/prompt",
    "params": {
      "sessionId": "abc-123",
      "prompt": [
        { "type": "text", "text": "请分析这个仓库的架构模式" }
      ]
    }
  }'
```

### 6.4 通过 CLI

```bash
# 一次性提示词
routa -p "Explain the architecture of this repository"

# 交互式会话
routa -p "Plan the next refactor for this codebase"

# 查看会话列表
routa acp list
routa acp runtime-status
```

### 6.5 ROUTA 协调者工作流

在 Session 模式中使用 ROUTA 角色时，遵循以下标准工作流：

```
1. 理解需求 → 询问澄清问题（1-4个）
2. 编写规格 → 使用 set_note_content 创建 @@task 块
3. 等待审批 → 向用户展示计划，等待确认
4. 委派 Wave 1 → delegate_task_to_agent(taskId, specialist="CRAFTER")
5. 结束回合 → 等待 Wave 1 完成
6. 验证 → 委派 GATE 智能体验证
7. 迭代 → 如有问题创建修复任务，重新委派
8. 完成 → 更新规格，记录结果
```

## 7. 模式 B：Kanban 看板自动化模式

Kanban 模式是 Routa 最强大的协作模式，将看板作为协调总线，每个泳道由不同专家智能体负责。

### 7.1 核心原理

```
用户需求 → Workspace + Kanban Board
                                ↓
Backlog → Todo → Dev → Review → Done
Refiner  Orch.  Crafter  Guard  Reporter
                         ↘
                      Blocked Resolver
```

**关键设计**：每个下游泳道比上游更严格——不信任上游输出，独立验证后才执行自己的工作。

### 7.2 创建看板

#### 通过 UI

1. 工作区页面 → Kanban 面板
2. 系统会自动创建默认看板（含 Backlog/Todo/Dev/Review/Done/Blocked 泳道）
3. 可自定义泳道和自动化配置

#### 通过 API

```bash
# 获取看板列表（含自动创建默认看板）
curl "http://localhost:3000/api/kanban/boards?workspaceId=your-workspace-id"

# 创建自定义看板
curl -X POST http://localhost:3000/api/kanban/boards \
  -H "Content-Type: application/json" \
  -d '{
    "workspaceId": "your-workspace-id",
    "name": "Sprint 1",
    "isDefault": true
  }'
```

### 7.3 添加卡片并触发自动化

```bash
# 创建卡片（任务）
curl -X POST http://localhost:3000/api/tasks \
  -H "Content-Type: application/json" \
  -d '{
    "workspaceId": "your-workspace-id",
    "boardId": "board-id",
    "title": "实现用户认证系统",
    "description": "包含登录、注册和密码重置功能",
    "columnId": "backlog",
    "priority": "high"
  }'
```

### 7.4 自动化触发流程

当卡片移动到开启了自动化的泳道时，系统自动触发：

```
1. 卡片移动到 column（automation.enabled=true 的列）
   ↓
2. column-transition.ts 发出 COLUMN_TRANSITION 事件
   ↓
3. workflow-orchestrator.ts 接收事件，通过 agent-trigger.ts 构建任务提示词
   ↓
4. 会话加入 kanban-session-queue.ts 队列，遵守看板并发限制（默认1）
   ↓
5. 创建 ACP 会话，分配对应泳道专家
   ↓
6. 专家完成工作后，调用 move_card 工具移动卡片到下一列
   ↓
7. 下一列的专家自动接管
```

### 7.5 各泳道专家的职责与门禁

#### Backlog Refiner（需求澄清）

- **职责**：澄清需求，输出 canonical YAML story block
- **门禁**：必须包含完整的 YAML story，含 INVEST 检查
- **可用工具**：`update_task`, `update_card`, `search_cards`, `create_card`, `decompose_tasks`, `move_card`
- **不可做**：不实现代码，不运行命令，不创建 GitHub Issue

#### Todo Orchestrator（执行编排）

- **职责**：不信任 Backlog 输出，重新解析 YAML，追加执行简报
- **入口门禁**：必须存在可解析的 canonical YAML story block
- **输出**：Execution Plan, Key Files & Entry Points, Dependency Plan, Risk Notes

#### Dev Crafter（开发实现）

- **职责**：不信任计划，验证可执行后才编码，实现+提交+记录 Dev Evidence
- **入口门禁**：YAML story + AC + Execution Plan + Key Files + Dependency Plan
- **输出**：Dev Evidence（changed files, tests run, per-AC verification, caveats）
- **安全规则**：
  - 不假设 `http://localhost:3000` 是正确的预览目标
  - 不使用 `pkill -f "next dev"` 等宽泛的进程终止命令
  - 只停止本会话启动的精确进程

#### Review Guard（代码审查）

- **职责**：独立验证每个 AC，要求测试和干净 git 状态
- **入口门禁**：Dev Evidence 存在且包含 per-AC verification
- **拒绝条件**：缺失 AC 验证、无测试、范围蔓延、lint 失败、文件预算违规
- **裁决**：`APPROVED` 或 `REJECTED`（回退到 Dev）

#### Done Reporter（完成记录）

- **职责**：追加完成摘要
- **入口门禁**：Review Findings 含 APPROVED 裁决
- **规则**：不可再移动卡片

#### Blocked Resolver（阻塞解决）

- **职责**：分类阻塞类型，说明根因，路由回正确泳道
- **输出**：Blocker Analysis（type, root cause, resolution, routing decision）

### 7.6 卡片工件随泳道演进

同一张卡片随工作推进会越来越严格：

| 泳道 | 追加的内容 |
|------|-----------|
| Backlog | canonical YAML story block |
| Todo | Execution Plan, Key Files, Dependency Plan, Risk Notes |
| Dev | Dev Evidence（changed files, tests, AC verification） |
| Review | Review Findings（verdict, per-AC status, issues） |
| Done | Completion Summary |

### 7.7 看板并发控制与监督

- **并发限制**：每个看板默认同时只运行1个自动化会话（可配置）
- **Dev 会话监督**：支持两种模式
  - `watchdog_retry`：超时后自动重试
  - `ralph_loop`：完成后检查是否满足完成条件，不满足则继续
- **自动推进**：`autoAdvanceOnSuccess=true` 时，成功完成后自动移动到下一列
- **非 Dev 泳道循环保护**：最多自动重试3次，防止无限循环

### 7.8 泳道间协作（Lane Handoff）

下游泳道可向上游泳道请求支持：

```bash
# Review 泳道请求 Dev 泳道准备运行环境
# 通过 request_previous_lane_handoff 工具

# Dev 泳道完成后通过 submit_lane_handoff 响应
```

## 8. 模式 C：YAML Workflow 工作流模式

Workflow 模式适合定义可重复的多步骤自动化流水线。

### 8.1 工作流定义格式

```yaml
name: "工作流名称"
description: "工作流描述"
version: "1.0"

trigger:
  type: manual | webhook | schedule
  source: github        # webhook 专用
  event: pull_request   # webhook 专用
  cron: "0 9 * * 1-5"   # schedule 专用

variables:
  model: "${ANTHROPIC_MODEL:-claude-sonnet-4-20250514}"
  key: "${ENV_VAR:-default_value}"

steps:
  - name: "步骤名称"
    specialist: "specialist-id"       # 引用 specialist YAML
    adapter: "claude-code-sdk"        # 适配器类型
    config:
      model: "${model}"
      max_turns: 50
      cwd: "/path/to/repo"
    input: |
      你的提示词内容...
      ${trigger.payload}             # 引用触发负载
      ${variables.key}               # 引用变量
      ${steps.上一步名称.output}      # 引用上一步输出
    output_key: "step_output_key"    # 输出键名
    parallel_group: "group-a"        # 并行组
    on_failure: stop | continue | retry
    max_retries: 3
    timeout_secs: 300
```

### 8.2 预置工作流

| 工作流 | 文件 | 说明 |
|--------|------|------|
| SDLC Flow | `resources/flows/sdlc.yaml` | 需求澄清 → 实现规划 → 计划审查 |
| Code Review Flow | `resources/flows/code-review.yaml` | 上下文收集 → Diff 分析 → 误报过滤 → 最终报告 |
| PR Verification Flow | `resources/flows/pr-verify.yaml` | 需求分析 → Review 分析 → 构建检查 → 最终裁决 |
| Simple Dev Flow | `resources/flows/simple-dev.yaml` | 简单的单步开发流程 |

### 8.3 触发工作流

#### 通过 API

```bash
# 手动触发
curl -X POST http://localhost:3000/api/workflows/sdlc/trigger \
  -H "Content-Type: application/json" \
  -d '{
    "workspaceId": "your-workspace-id",
    "triggerPayload": "实现用户认证系统，包含登录、注册和密码重置"
  }'
```

#### 通过 CLI

```bash
routa workflow run resources/flows/sdlc.yaml \
  --trigger-payload "Add a login page with OAuth support"
```

### 8.4 创建自定义工作流

```bash
# 通过 API 创建
curl -X POST http://localhost:3000/api/workflows \
  -H "Content-Type: application/json" \
  -d '{
    "id": "my-custom-flow",
    "yamlContent": "name: My Custom Flow\nsteps:\n  - name: Step 1\n    specialist: developer\n    input: Do something"
  }'
```

### 8.5 工作流执行引擎

```
WorkflowLoader.load() → WorkflowDefinition
    ↓
WorkflowExecutor.trigger() → 创建 WorkflowRun + BackgroundTasks
    ↓
BackgroundWorker.dispatchPending() → 按依赖图执行各步骤
    ↓
步骤间通过 output_key 和 ${steps.X.output} 传递数据
    ↓
同 parallel_group 的步骤并发执行
```

## 12. A2A 跨智能体互操作

A2A（Agent-to-Agent）协议允许 Routa 与外部智能体系统互操作。

### 12.1 A2A 架构

```
Routa 看板 → KanbanAutomationStep(transport: "a2a")
    ↓
A2A Outbound Client → 发送消息到外部 Agent Card URL
    ↓
外部智能体处理任务
    ↓
waitForCompletion() → 接收结果
    ↓
发出 AGENT_COMPLETED/AGENT_FAILED 事件
```

### 12.2 配置 A2A 自动化

在看板泳道自动化步骤中指定 `transport: "a2a"` 和 `agentCardUrl`：

```yaml
# 看板自动化步骤配置示例
steps:
  - specialistId: "external-reviewer"
    specialistName: "External Reviewer"
    role: "GATE"
    transport: "a2a"
    agentCardUrl: "https://external-agent.example.com/card"
    authConfigId: "auth-config-1"
```

### 12.3 A2A API

```bash
# 获取 Agent Card
curl http://localhost:3000/api/a2a/card

# 发送消息
curl -X POST http://localhost:3000/api/a2a/message \
  -H "Content-Type: application/json" \
  -d '{
    "agentCardUrl": "https://external-agent.example.com/card",
    "message": "请审查这个 PR",
    "metadata": { "workspaceId": "...", "cardId": "..." }
  }'

# 查询任务状态
curl http://localhost:3000/api/a2a/tasks/[taskId]
```

## 17. 完整端到端实战示例

### 场景：使用 Kanban 看板自动完成"构建用户认证系统"

#### Step 1：环境准备

```bash
# 安装 CLI
npm install -g routa-cli

# 或启动 Web
npm install --legacy-peer-deps && npm run dev
```

#### Step 2：创建工作区并配置

```bash
# 创建工作区
curl -X POST http://localhost:3000/api/workspaces \
  -d '{"name": "Auth System Project"}'

# 附加代码库
curl -X POST http://localhost:3000/api/workspaces/[wsId]/codebases \
  -d '{"path": "/my-project", "branch": "main"}'
```

#### Step 3：打开 Kanban 看板

在 UI 中导航到 `/workspace/[wsId]/kanban`，系统自动创建默认看板。

#### Step 4：创建需求卡片

在 Backlog 列创建卡片：

```
标题：实现用户认证系统
描述：构建完整的用户认证系统，包含登录、注册和密码重置功能
优先级：high
标签：auth, security
```

#### Step 5：自动化流转

```
Backlog Refiner 自动接管
  → 澄清需求，输出 canonical YAML story
  → 包含 AC：AC1=登录功能, AC2=注册功能, AC3=密码重置
  → INVEST 检查通过
  → move_card → Todo

Todo Orchestrator 自动接管
  → 重新解析 YAML，不信任 Backlog 输出
  → 追加：Execution Plan, Key Files, Dependency Plan
  → move_card → Dev

Dev Crafter 自动接管
  → 验证 YAML story + AC + Execution Plan 完整
  → 实现登录、注册、密码重置
  → 运行测试，提交代码
  → 记录 Dev Evidence
  → git status 确认干净
  → move_card → Review

Review Guard 自动接管
  → 独立验证 AC1/AC2/AC3
  → 检查测试覆盖
  → 检查 git 状态
  → 检查 lint
  → APPROVED → move_card → Done

Done Reporter 自动接管
  → 追加 Completion Summary
  → 卡片完成
```

#### Step 6：查看追踪

```bash
# 查看完整执行追踪
curl http://localhost:3000/api/traces?workspaceId=[wsId]

# 在 UI 中查看
# 导航到 /traces 页面
```

## 19. 故障排查指南

### 19.1 Provider 无法连接

```bash
# 检查运行时状态
curl http://localhost:3000/api/acp/registry
curl http://localhost:3000/api/acp/runtime

# 检查 Docker 容器状态
curl http://localhost:3000/api/acp/docker/status
```

### 19.2 看板自动化不触发

1. 检查泳道的 `automation.enabled` 是否为 `true`
2. 检查 `transitionType` 是否正确（`entry` / `exit` / `both`）
3. 检查并发限制是否已满（看板队列）
4. 检查 `ROUTA_INTERNAL_API_ORIGIN` 是否正确配置

### 19.3 智能体会话超时

1. 检查 Dev 会话监督配置
2. 检查 `inactivityTimeoutMinutes` 设置
3. 查看 Watchdog 日志

### 19.4 追踪文件丢失

1. 检查 JSONL 文件路径权限
2. 检查 `storage/` 目录是否存在
3. 使用 `TraceReader` 查询

### 19.5 质量门禁失败

```bash
# 运行 Fitness 检查
cargo build -p entrix
entrix run --tier fast

# 检查文件预算
# 查看 docs/fitness/file_budgets.json
```
