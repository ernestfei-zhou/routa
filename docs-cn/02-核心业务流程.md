# Routa.js 核心业务流程详解

> 版本：0.17.0 | 本文档详细描述 Routa.js 平台支持的十大核心业务流程

---

## 流程一：Agent 会话管理（ACP 工作流）

ACP（Agent Chat Protocol）是 Routa.js 最核心的协议，管理 Agent 进程的完整生命周期。

### 流程图

```text
创建 Agent 记录
      │
      ▼
安装 ACP 二进制 ─── 检测本地二进制 / Docker 镜像
      │
      ▼
配置 MCP 服务 ─── 生成 MCP 配置文件（.routa/mcp.json）
      │
      ▼
热身进程（Warmup）─── 预启动 Agent 进程以减少首次响应延迟
      │
      ▼
发起会话（Prompt）─── 发送任务提示到 Agent 进程
      │
      ▼
流式响应（SSE）─── 实时推送 Agent 输出到前端
      │
      ▼
持久化历史 ─── 写入数据库 + JSONL 追踪文件
      │
      ▼
会话完成/断开 ─── 更新 Agent 状态、清理进程
```

### 支持的执行后端

| 后端 | 说明 | 适配器 |
|------|------|--------|
| **Claude Code SDK** | Anthropic 官方 SDK 集成 | `claude-code-sdk-adapter.ts` / `claude-code-process.ts` |
| **OpenCode SDK** | OpenCode AI SDK 集成 | `opencode-sdk-adapter.ts` / `opencode-process.ts` |
| **Docker** | Docker 容器化隔离执行 | `docker-opencode-adapter.ts` / `docker/process-manager.ts` |
| **HTTP API** | 基于 HTTP 的 API 提供者 | `api-based-providers.ts` |
| **标准 ACP** | 兼容 ACP 协议的 CLI | `standard-acp-adapter.ts` |
| **Workspace Agent** | 内置工作区 Agent | `workspace-agent-adapter.ts` |

### Provider 归一化模式

```text
Provider 进程/桥接
  → Provider 特定输出/通知
  → 适配器归一化（Adapter Normalization）
  → 统一会话更新（Unified Session Updates）
  → 持久化、追踪和 UI 流式
```

## 流程二：多 Agent 编排（Orchestration）

父 Agent 收到任务后，可以创建子 Agent 进行任务委派，实现多层级协作。

### 流程图

```text
父 Agent 收到任务
      │
      ▼
解析任务块（Task Block Parser）─── 从 Prompt 中提取子任务
      │
      ▼
创建子 Agent 记录 ─── 设置 role/model_tier/workspace_id/parent_id
      │
      ▼
启动 ACP 进程 ─── 为子 Agent 创建独立进程
      │
      ▼
发送任务 Prompt ─── 包含上下文和具体指令
      │
      ▼
监听完成事件 ─── EventBus 等待 AgentCompleted/AgentError
      │
      ▼
子 Agent 回报 ─── 结果写入 Artifact 或更新 Task
      │
      ▼
唤醒父 Agent ─── 父 Agent 整合子 Agent 结果
```

### 委派深度控制

`delegation-depth.ts` 管理委派层级，防止无限递归：
- 最大深度限制
- 循环检测
- 跨工作区委派校验

## 流程三：看板自动化（Kanban Automation）

看板不仅仅是 UI 投影，它还驱动基于列的自动化和队列化。这是 Routa.js 区别于普通看板工具的核心特性。

### 流程图

```text
卡片移入自动化列（如 "In Progress"）
      │
      ▼
检测列自动化配置 ─── 检查 specialist / adapter / required_artifacts
      │
      ▼
任务合同就绪检查 ─── 验证 required_task_fields 是否完整
      │
      ▼
┌─ ACP 传输 ─────────────────────────────────┐
│  触发 Specialist/Agent 会话                  │
│  创建 ACP Session                            │
│  发送 Prompt（含任务上下文 + 工作区信息）    │
└─────────────────────────────────────────────┘
      │
      ▼
会话完成 ─── 检查 required_artifacts 是否已产出
      │
      ▼
交付就绪检查 ─── task-delivery-readiness.ts
      │
      ▼
自动推进卡片到下一列 ─── 基于 transition policy
      │
      ▼
看板事件广播 ─── SSE 通知前端刷新
```

### 关键子模块

| 模块 | 职责 |
|------|------|
| `agent-trigger.ts` | 判断是否触发 Agent 会话，选择 Specialist |
| `kanban-session-queue.ts` | 看板级并发控制，防止过期条目重复触发 |
| `workflow-orchestrator.ts` | 多列工作流编排，处理列间转移 |
| `task-delivery-readiness.ts` | 交付门检查：工件是否齐备、字段是否完整 |
| `column-transition.ts` | 列转移策略和验证 |
| `restart-recovery.ts` | 服务重启后的看板状态恢复 |

### 两种传输协议

看板自动化支持两种传输协议：
- **ACP**：本地 Agent 进程交互
- **A2A**：远程 Agent 系统互操作

## 流程四：YAML 工作流引擎（Workflow Engine）

YAML 驱动的多步骤 Agent 工作流，定义在 `resources/flows/` 目录下。

### 流程图

```text
加载 workflow.yaml
      │
      ▼
解析 WorkflowDefinition ─── name / trigger / variables / steps
      │
      ▼
步骤执行循环：
  ┌───────────────────────────────────────┐
  │  解析当前步骤                          │
  │       │                                │
  │       ▼                                │
  │  变量替换 ─── ${model} / ${steps.*}    │
  │       │                                │
  │       ▼                                │
  │  调用 Specialist ─── 通过 AcpAgentCaller│
  │       │                                │
  │       ▼                                │
  │  收集输出 ─── 存入 output_key          │
  │       │                                │
  │       ▼                                │
  │  失败处理 ─── OnFailure 策略           │
  └───────────────────────────────────────┘
      │
      ▼
工作流完成/失败
```

### 预定义工作流

| 工作流 | 文件 | 说明 |
|--------|------|------|
| **SDLC** | `sdlc.yaml` | 端到端软件开发生命周期 |
| **代码审查** | `code-review.yaml` | 自动化代码审查流程 |
| **PR 验证** | `pr-verify.yaml` | PR 合并前验证 |
| **简单开发** | `simple-dev.yaml` | 基础开发任务流程 |

### WorkflowDefinition 结构

```yaml
name: "SDLC Flow"
description: "端到端软件开发生命周期"
version: "1.0"

trigger:
  type: manual          # manual | webhook | schedule

variables:
  model: "GLM-4.7"
  base_url: "${ANTHROPIC_BASE_URL}"

steps:
  - name: "需求精化"
    specialist: "issue-refiner"
    adapter: "claude-code-sdk"
    config:
      model: "${model}"
    input: "${trigger.payload}"
    output_key: "refined_requirements"

  - name: "实施计划"
    specialist: "routa"
    input: "${steps.需求精化.output}"
    output_key: "implementation_plan"
```

## 流程五：项目健康度分析（Fitness）

从代码库中提取结构化信号，评估项目可治理性和代码质量。

### 流程图

```text
扫描代码库
      │
      ▼
提取 Harness 信号 ─── package.json / 脚本入口 / 构建配置 / CI 配置
      │
      ▼
规范检测 ─── spec_detector 自动识别技术栈
      │
      ▼
架构质量分析 ─── 模块依赖图 / 分层合规性
      │
      ▼
生成 Fitness 报告 ─── harnessability 基线 / 架构评分
      │
      ▼
运行 Entrix 检查 ─── docs/fitness/ 规则集
      │
      ▼
生成可执行计划 ─── fitness/plan
```

### Harness 四层循环

```text
Context（上下文）→ Run（运行）→ Observe（观察）→ Govern（治理）
     ↑                                              │
     └──────────────────────────────────────────────┘
```

核心记录类型：`Task / Run / Workspace / EvalSnapshot / PolicyDecision / Evidence`

## 流程六：Agent 追踪（Trace）

追踪记录 Agent 的操作影响，是一等调试和归因机制。

### 流程图

```text
Agent 操作工具（Tool Call）
      │
      ▼
提取文件变更 ─── file-range-extractor 识别影响的文件和行范围
      │
      ▼
记录 TraceRecord ─── session / tool / file / range / model
      │
      ▼
JSONL 存储 ─── 追加写入 JSONL 文件
      │
      ▼
VCS 上下文关联 ─── 关联 Git commit / branch / diff
      │
      ▼
追踪查询与回放 ─── 支持按会话/文件/时间范围查询
```

### 追踪数据结构

- **TraceRecord**：会话 ID、工具名、文件路径、行范围、模型信息
- **VCS 上下文**：Git commit SHA、分支名、变更差异
- **存储格式**：JSONL 文件 + 数据库索引

## 流程七：沙箱执行（Sandbox）

Docker 隔离的代码执行环境，支持策略化的安全控制。

### 流程图

```text
创建沙箱请求
      │
      ▼
策略解析 ─── 网络策略 / 挂载策略 / 权限约束
      │
      ▼
创建 Docker 容器 ─── 应用沙箱策略
      │
      ▼
关联 Worktree ─── 挂载代码到容器
      │
      ▼
代理执行请求 ─── 在容器内执行代码
      │
      ▼
流式输出 ─── NDJSON 格式返回结果
      │
      ▼
销毁容器 ─── 清理资源
```

### 策略维度

- **网络策略**：允许/禁止的网络访问
- **挂载策略**：可挂载的目录和文件
- **权限约束**：可执行的操作限制

## 流程八：定时调度（Scheduling）

基于 Cron 表达式的定时 Agent 触发机制。

### 流程图

```text
定义调度规则 ─── Cron 表达式 + Agent + Prompt
      │
      ▼
注册到 ScheduleStore
      │
      ▼
Tick 轮询 ─── 定时检查是否有需要触发的调度
      │
      ▼
触发调度 ─── 创建 Task + 启动 ACP 会话
      │
      ▼
记录执行历史 ─── last_run_at / next_run_at
```

### 调度触发方式

- **Cron 表达式**：标准 Cron 格式定义触发时间
- **手动触发**：`POST /api/schedules/{id}/run`
- **Webhook 触发**：GitHub 事件等外部触发
- **轮询触发**：`polling/github-polling-adapter.ts` 定期检查

## 流程九：技能发现（Skills）

从项目目录中自动发现和注册 Agent 可用的技能。

### 流程图

```text
扫描技能目录 ─── .opencode/skills / .claude/skills / .agents/skills
      │
      ▼
解析 SKILL.md ─── YAML frontmatter + Markdown 正文
      │
      ▼
注册到 SkillRegistry ─── 名称 / 描述 / 入口路径
      │
      ▼
Agent 可引用技能 ─── 通过 MCP 工具或直接 Prompt
```

### 技能发现路径

1. 项目本地目录：`.opencode/skills/`、`.claude/skills/`、`.agents/skills/`
2. 仓库克隆：通过 `/api/skills/clone` 克隆技能仓库
3. 上传：通过 `/api/skills/upload` 上传 ZIP 包

## 流程十：跨 Agent 互操作（A2A）

实现不同 Agent 系统之间的互操作，遵循 A2A 协议规范。

### 流程图

```text
发现远程 Agent Card ─── GET /api/a2a/card
      │
      ▼
建立 A2A 会话 ─── 通过 OutboundClient 连接
      │
      ▼
发送任务 ─── POST /api/a2a/message 或 JSON-RPC
      │
      ▼
接收结果 ─── SSE 流式或 JSON-RPC 响应
      │
      ▼
任务桥接 ─── A2aTaskBridge 将 A2A 任务映射为本地 Task
      │
      ▼
会话注册 ─── A2aSessionRegistry 管理活跃会话
```

### A2A 核心组件

| 组件 | 职责 |
|------|------|
| `A2aAgentCard` | Agent 身份和能力声明 |
| `A2aOutboundClient` | 出站连接客户端 |
| `A2aSessionRegistry` | 活跃会话管理 |
| `A2aTaskBridge` | A2A 任务到本地 Task 的桥接 |
| `A2aAuthConfig` | 认证配置管理 |

## 流程间协作关系

```text
                    ┌────────────────────────┐
                    │      Workspace         │
                    │   (顶级协调边界)        │
                    └───────────┬────────────┘
                                │
        ┌───────────┬───────────┼───────────┬───────────┐
        │           │           │           │           │
   ┌────▼────┐ ┌────▼────┐ ┌───▼────┐ ┌────▼────┐ ┌────▼────┐
   │  ACP    │ │ Kanban  │ │Workflow│ │Schedule │ │  A2A    │
   │ 会话管理│ │ 自动化  │ │ 引擎   │ │ 定时调度│ │ 互操作  │
   └────┬────┘ └────┬────┘ └───┬────┘ └────┬────┘ └────┬────┘
        │           │          │            │           │
        └───────┬───┴──────────┴──────┬─────┴───────────┘
                │                      │
          ┌─────▼──────┐        ┌──────▼──────┐
          │ Orchestration│      │   Trace     │
          │  多Agent编排 │      │  追踪归因    │
          └─────────────┘      └─────────────┘
                │                      │
          ┌─────▼──────┐        ┌──────▼──────┐
          │  Sandbox   │        │   Fitness   │
          │  沙箱执行  │        │  健康度分析  │
          └────────────┘        └─────────────┘
```

**关键交互点**：
- 看板自动化 → ACP 会话（列转移触发 Agent）
- 工作流引擎 → ACP 会话（步骤执行调用 Specialist）
- 定时调度 → ACP 会话（定时触发 Agent）
- A2A 互操作 → 看板任务桥接
- 编排 → ACP 会话（子 Agent 派生）
- 所有 Agent 执行 → Trace 追踪
- Fitness 分析 → Harness 信号提取
